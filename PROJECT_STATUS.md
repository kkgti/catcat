# 项目现状梳理

## 项目定位

本项目是一个“找伪装猫”的抖音小程序游戏原型，当前代码库同时包含：

- 小程序正式实现
- 浏览器预览/验收版本

其中：

- 小程序代码位于项目根目录、`pages/`、`utils/`
- `preview.html` 是浏览器端的完整预览版本，可作为功能验收和对照基准

## 当前实现形态

### 1. 小程序端

小程序端不是常规的组件化游戏页面，而是：

- 首页：负责关卡列表、成就列表、图鉴入口、进度展示
- 游戏页：只承载一个 `canvas`
- 游戏核心逻辑：集中在 `utils/game-engine.js`

这意味着当前项目属于“Canvas 驱动的轻量游戏架构”。

### 2. 预览端

`preview.html` 是浏览器可运行版本，包含完整的页面、Canvas、音效、存档和交互逻辑。现有多个 `utils` 文件顶部都注明“同步自 `preview.html`”，说明它是当前小程序版的重要来源和验收参考。

## 当前核心玩法

玩家进入房间后，在大量物品中寻找“伪装成家具/道具/杂物的猫”。

基础规则：

- 每关有固定数量的猫需要找出
- 点错会扣生命
- 找完全部猫则通关
- 通关后可获得星级，并解锁后续关卡

## 当前主要问题

### 1. 终端查看时存在乱码假象

当前已确认：

- 项目源码文件本身是 UTF-8 正常内容
- 在某些终端输出环境中，中文和 emoji 会显示为乱码
- 这属于查看环境的编码显示问题，不等于源码已经损坏

因此现阶段不应把“批量修乱码”作为首要任务，后续应优先在真实运行环境中验证：

- 小程序端页面实际显示是否正常
- `preview.html` 浏览器预览中的文案是否正常
- 是否存在个别文件被单独误存为错误编码

### 2. 小程序端与 `preview.html` 仍可能存在功能差异

虽然多个模块标注为“同步自 `preview.html`”，但从开发过程判断，小程序端很可能并不是完全 1:1 同步。

后续需要专门做一次：

- 功能对齐检查
- UI 表现差异检查
- 交互行为一致性检查

### 3. 音效尚未产品化

当前小程序端音效层仍偏接口占位，后续若要正式上线，需要补齐：

- 音频资源
- 抖音小程序实际播放策略
- 静音/恢复逻辑
- 生命周期处理

### 4. 上线质量层还未完整梳理

若目标是抖音小程序正式上线，当前还缺一轮系统检查，包括：

- 性能

## 玩法增强方向

当前版本已经具备完整基础玩法，但如果目标是做成更有记忆点、更适合抖音小程序传播的游戏，后续增强重点不应只是“继续加内容”，而应当让玩法本身更有个性。

### 一个更清晰的核心方向

建议将项目从：

- 找隐藏物

进一步升级为：

- 和会伪装、会反制、会互相配合的猫斗智

也就是说，后续不只是提升“眼力难度”，而是加强：

- 猫对玩家的反应
- 房间的规则差异
- 每局的戏剧性和变化感

### 当前玩法的优势

当前玩法已经有这些可继续放大的优点：

- “猫伪装成物品”这个主题本身有辨识度
- 房间主题已经存在，天然适合做机制区分
- 猫已有行为、表情、气泡、伙伴联动等基础系统
- 道具和环境事件已经接入，适合扩展成更深的博弈

### 当前玩法继续提升的关键

建议未来增强时，把重点放在这三类变化上：

1. 房间不是只换背景，而是换规则
2. 猫不是只会轻微动，而是会骗、会躲、会联动
3. 玩家不是只负责点击，而是要观察、判断、取舍

## 建议优先强化的三条主线

### 1. 房间机制差异化

让每个房间都拥有一句话能说清楚的核心规则。

示例方向：

- 客厅：真假动静混杂，训练玩家观察“什么是猫的动静”
- 厨房：蒸汽、热气、锅具声造成遮挡和误导
- 卧室：灯光变暗，更多依赖轮廓、反光和微动作
- 浴室：镜像和水汽制造假线索
- 书房：书堆位移，部分猫可能换位或换伪装

这样玩家记住的就不只是“第几关更难”，而是“这个房间是怎么骗我的”。

### 2. 特殊猫类型

建议在现有普通猫基础上，逐步加入少量高辨识度的特殊猫。

优先推荐：

- 伪线索猫：故意制造错误动静，引导玩家点错
- 搭档猫：一只暴露时，另一只会联动
- 换位猫：经过一段时间后重新伪装或换位置

后续可扩展：

- 胆小猫：玩家靠近或频繁点击时会进入“完全静止”
- 调皮猫：会在玩家视线外制造额外误导
- 领袖猫：存在时会提升全场猫的表现欲或干扰频率

### 3. 玩家与猫的博弈感

建议增强“玩家操作会影响猫”的反馈，而不只是单向找目标。

示例方向：

- 玩家点错太多，猫会提高警觉
- 玩家长期盯一个区域，附近猫会暂时收敛动作
- 玩家连续使用某种道具，部分猫会短时间适应
- 某些猫在玩家表现激进时反而更难暴露

这一类机制会让游戏从“找东西”升级成“和猫对抗”。

## 建议新增的高潜力机制

### 1. 警觉值系统

为每局加入全局或局部的“猫咪警觉值”。

可能来源：

- 连续点错
- 高频使用道具
- 短时间连续点击

可能效果：

- 猫减少动作频率
- 伪线索增多
- 真猫更晚露出破绽

意义：

- 增加操作取舍
- 防止无脑乱点
- 强化玩家与猫的对抗感

### 2. 房间事件规则化

目前已有环境事件，后续可将其从“随机氛围事件”升级为“可影响判断的玩法事件”。

示例：

- 厨房蒸汽短时间遮住部分区域
- 卧室关灯后只剩眼睛反光
- 浴室镜面会延迟反射动作
- 书房纸页飞动时掩盖真实抖动

### 3. 猫关系系统

不仅让猫单独存在，还让它们之间产生关系。

示例：

- 搭档关系
- 亲子关系
- 模仿关系
- 领袖关系

意义：

- 增强世界感
- 让“找到一只猫”引发新的局面变化
- 提高局内戏剧性

### 4. 局内故事化主题

不做重剧情，而是做“每局一句话主题”。

示例：

- 偷吃失败的厨房局
- 雨夜卧室局
- 洗澡日浴室局
- 读书会书房局

作用：

- 让同一房间可以衍生多局不同体验
- 方便做活动主题和节日内容

### 5. 判断型目标

后续可让部分关卡不只是“全部找出”，而是加轻度判断层。

示例：

- 先找出最可疑的一只
- 找出猫王后触发全场变化
- 可先标记怀疑目标，再确认
- 某些误点不会扣血，但会提升警觉值

## 最推荐的第一轮玩法升级方案

如果只做一轮最值的增强，建议优先实现下面这组组合：

1. 房间专属机制
2. 3 种特殊猫类型
3. 轻量警觉值系统

原因：

- 最能拉开与普通找茬/找不同的差异
- 最适合短局小程序游戏
- 和现有架构最兼容
- 后续容易继续叠加图鉴、成就、挑战玩法

## 第一轮可落地的设计建议

### 方案 A：先做“房间机制”

适合快速见效。

建议：

- 厨房加入蒸汽遮挡
- 卧室加入暗灯观察
- 浴室加入镜像误导

优点：

- 不一定需要大改底层结构
- 玩家体感变化明显

### 方案 B：先做“特殊猫”

适合强化主题表达。

建议：

- 伪线索猫
- 搭档猫
- 换位猫

优点：

- 最贴合“变身躲猫猫”的核心设定
- 对传播和记忆点帮助大

### 方案 C：先做“警觉值”

适合增强博弈感。

建议：

- 将误点、乱点、连续用道具纳入风险
- 警觉值高时，猫更安静、假线索更多

优点：

- 直接改变玩家策略
- 让玩法更有深度

## 当前推荐的实际开发顺序

若从开发投入与效果平衡考虑，建议按这个顺序推进：

1. 先做 1 个房间机制样板
2. 再做 1 种特殊猫类型
3. 最后补一个轻量警觉值系统

推荐首批组合：

- 厨房蒸汽遮挡
- 搭档猫
- 全局警觉值

理由：

- 最容易让玩家立刻感受到“这游戏不只是找东西”
- 和当前系统兼容性较高
- 做出来后可以快速验证手感

## 后续规划架构

考虑到项目后续会持续迭代，而单次协作的上下文和记忆有限，建议把后续工作固定在一套更稳定的规划架构里。后面所有新增功能，尽量都落到这几个长期模块下面，不再零散推进。

### 一. 核心玩法层

这是游戏最重要的一层，决定游戏是否“好玩”和“有记忆点”。

包含：

- 找猫主循环
- 点击判断与惩罚
- 猫的伪装、行为、反制
- 房间规则差异
- 局内目标和挑战目标

这层的目标不是增加数量，而是提升：

- 判断乐趣
- 变化感
- 博弈感

### 二. 内容生产层

这是负责“持续出新内容”的层，决定游戏能不能越做越丰富。

包含：

- 新房间
- 新伪装模板
- 新特殊猫类型
- 新环境事件
- 新主题局
- 新图鉴内容

这一层要尽量做成可复用、可组合，而不是每次都写死一整套新逻辑。

### 三. 成长反馈层

这是让玩家持续玩下去的层，负责给玩家长期目标。

包含：

- 成就
- 图鉴
- 星级
- 连击表现
- 结算展示
- 关卡解锁

后续也可以扩展到：

- 每日挑战
- 隐藏收集
- 特殊猫收录
- 高分/无伤/速通记录

### 四. 表现与氛围层

这是让游戏“有感觉”的层，负责视觉、音效、情绪和戏剧性。

包含：

- 动画
- 粒子效果
- 表情系统
- 环境事件表现
- 背景氛围
- 音效与反馈

这一层通常不改变规则，但会显著提高体感和品质感。

### 五. 平台上线层

这是面向抖音小程序落地的层，负责保证游戏能稳定上线。

包含：

- 权限
- 分享链路
- 存档稳定性
- 平台兼容
- 性能优化
- 异常容错
- 页面与入口整合

这一层不直接决定玩法，但会决定项目能不能真正上线。

## 建议的开发主线

后续开发建议始终沿着下面这条顺序推进：

1. 先强化核心玩法层
2. 再补内容生产层
3. 同步维护成长反馈层
4. 然后提升表现与氛围层
5. 在合适时机集中处理平台上线层

这样做的原因是：

- 如果先做平台层，游戏可能能跑，但不够好玩
- 如果只加内容不改玩法，会很快同质化
- 如果玩法先立住，后面的内容和上线工作才有意义

## 分阶段开发路线

下面是建议采用的长期开发阶段。后续每次开发，都尽量只聚焦当前阶段，不跨太多层一起做。

### Phase 1：玩法立骨架

目标：

- 明确游戏要变成什么样
- 把“找隐藏物”升级成“和猫斗智”
- 建立第一批真正有辨识度的机制

本阶段建议实现：

- 1 个房间专属机制样板
- 1 到 2 种特殊猫类型
- 1 个轻量警觉值系统

推荐组合：

- 厨房蒸汽遮挡
- 搭档猫
- 全局警觉值

阶段完成标准：

- 玩家能明显感受到玩法不再只是静态找猫
- 至少有一个房间拥有独特规则
- 至少有一种猫能形成“博弈感”

### Phase 2：玩法扩散到多个房间

目标：

- 把第一阶段的玩法升级从“样板”扩展到“体系”
- 让不同房间真正拥有差异化体验

本阶段建议实现：

- 为 3 到 5 个房间分别配一个核心机制
- 增加更多特殊猫类型
- 调整不同房间的规则组合和难度节奏

示例：

- 客厅：真假动静
- 厨房：蒸汽遮挡
- 卧室：暗灯观察
- 浴室：镜像误导
- 书房：换位/换伪装

阶段完成标准：

- 玩家能明显区分各房间玩法
- 房间不再只是换皮和数值变化

### Phase 3：内容与成长闭环

目标：

- 让玩家不仅“能玩”，还能“想继续玩”

本阶段建议实现：

- 特殊猫图鉴或特殊图鉴标签
- 新成就与挑战目标
- 更丰富的结算反馈
- 主题局或变种局

阶段完成标准：

- 玩法升级后的内容有成长承载
- 玩家在通关之外还有额外目标

### Phase 4：表现升级

目标：

- 提高游戏辨识度和爽感
- 强化分享传播时的观感

本阶段建议实现：

- 关键机制的专属视觉表现
- 特殊猫登场/暴露反馈
- 更完整的音效系统
- 更强的房间氛围感

阶段完成标准：

- 关键玩法有明显视听反馈
- 游戏看起来更像成品而不是原型

### Phase 5：上线收口

目标：

- 面向抖音小程序正式上线做集中整理

本阶段建议实现：

- 统一入口和页面流程
- 检查分享、权限、相册、存档
- 做性能与兼容性检查
- 清理浏览器残留逻辑
- 补充上线前验收清单

阶段完成标准：

- 核心功能可稳定运行
- 关键平台能力已接通
- 具备提交审核和继续打磨的基础

## 当前建议先做的阶段

基于当前项目状态，建议立即进入：

- Phase 1：玩法立骨架

原因：

- 当前游戏已经“可玩”

## Phase 1 进一步拆分

为了便于短记忆协作，Phase 1 建议继续拆成更小的开发小节。

### Phase 1-1：设计样板机制

目标：

- 先确定第一批最值得做的机制
- 不急着大面积铺开

建议产出：

- 选定首个房间机制
- 选定首个特殊猫类型
- 选定警觉值的最小规则

### Phase 1-2：落地最小可玩版本

目标：

- 让第一批机制真正进游戏

建议产出：

- 样板机制可以在局内体验
- 与现有道具、提示、结算不冲突

### Phase 1-3：调参与验证

目标：

- 确认新机制不是只“有”，而是真的“好玩”

建议产出：

- 调整频率、数值、提示强度
- 调整是否过难、过乱、过于打断节奏

### Phase 1-4：固化为可复用结构

目标：

- 让后续房间和特殊猫扩展更容易

建议产出：

- 把样板逻辑整理成更可配置的结构
- 为 Phase 2 的扩展留接口

## 后续协作建议

为了适应长期开发，建议后续每次协作都按下面方式推进：

1. 先明确当前只做哪个 Phase
2. 再明确只做该 Phase 里的哪一个小节
3. 本轮只产出当前小节需要的内容
